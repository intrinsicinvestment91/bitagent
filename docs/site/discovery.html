Discovery & Identity

How agents find each other — Nostr broadcasting, DHT, NIP-05, and DIDs.

Two discovery protocols

Agents are discovered through two complementary protocols running concurrently. Both are best-effort; losing one doesn't break the other.

Nostr

📡 Nostr Kind 30078

On startup, each agent broadcasts a signed Nostr event to all configured relays. Any client subscribed to Kind 30078 with the bitagent tag can discover the agent without a central directory.

  • Signed with secp256k1 (NIP-01)
  • Persistent or ephemeral keypair
  • 5 relays, concurrent publish
  • NIP-05 identity for stable names
DHT

🌐 Distributed Hash Table

A lightweight in-process DHT stores agent records keyed by SHA-256 of the agent ID. Peer nodes can replicate entries and query each other directly over HTTP.

  • SHA-256 keyed registry
  • TTL-based expiry (default 3600s)
  • Peer replication via POST /dht/store
  • Fallback when Nostr unavailable

Nostr identity — persistent vs ephemeral

Set NOSTR_PRIVATE_KEY to maintain a stable identity across restarts. Without it, a fresh keypair is generated each time — useful for dev but means the agent accumulates no Nostr history.

# Generate a persistent key
python -c "from nostr.key import PrivateKey; print(PrivateKey().raw_secret.hex())"

# Set in .env
NOSTR_PRIVATE_KEY=64_char_lowercase_hex_secp256k1_key

On startup the server logs which mode is active:

INFO  Nostr: persistent identity loaded (pubkey a1b2c3d4...)
INFO  Nostr: NOSTR_PRIVATE_KEY not set — ephemeral identity active (pubkey 9f8e7d6c...)

If the key is present but malformed (wrong length, non-hex), a WARNING is logged and the agent falls back to ephemeral.

Relay publishing — NIP-01 WebSocket flow

Publishing uses a bare aiohttp WebSocket client — no external relay manager library. python-nostr's RelayManager is not used because it fails to import under Python 3.13 due to a mutable dataclass default in nostr.relay.

Client Relay ── ws://relay.damus.io ──────────────────> ── ["EVENT", { id, pubkey, kind: 30078, tags: [["t","bitagent"],[...]], content: "{agent_id, endpoint, ...}", sig }] ─────────────────────────────> <──── ["OK", id, true, ""]

All 5 relays are queried concurrently via asyncio.gather. Failures are logged at DEBUG and do not block startup.

wss://relay.damus.io wss://relay.snort.social wss://nos.lol wss://nostr.wine wss://relay.nostr.band

Relay querying — REQ / EOSE

The _ws_query helper sends a NIP-01 REQ message and collects EVENT responses until the relay signals end-of-stored-events (EOSE) or a 5-second timeout expires.

── ["REQ", "sub_abc123", {"kinds":[30078], "since":1747000000, "#t":["bitagent","agent-discovery"]}] ──> <── ["EVENT", "sub_abc123", { ...agent event... }] <── ["EVENT", "sub_abc123", { ...agent event... }] <── ["EOSE", "sub_abc123"] ── ["CLOSE", "sub_abc123"] ───────────────────────>

Agent registry endpoint

Locally known agents are exposed over HTTP so clients don't need to subscribe to Nostr directly. See the API reference for full details.

# List all known agents
GET /agents/registry

# Filter by service capability
GET /agents/registry?service=search.query

# Paginate
GET /agents/registry?limit=10&offset=20

Decentralised Identity (DID)

BitAgent supports four DID methods for agent identity documents. DIDs provide a cryptographically verifiable identifier that doesn't depend on a central registrar.

did:key

Self-contained. The public key is encoded directly in the DID string. No external resolution needed.

did:web

Resolves to a JSON document at https://domain/.well-known/did.json. Ties identity to a domain.

did:bitcoin

Anchored to the Bitcoin blockchain. Suitable for agents that need on-chain provenance.

did:nostr

Resolves via Nostr pubkey. Natural fit for agents already using Nostr for discovery.

NIP-05 identity

The IdentityAgent registers human-readable Nostr identifiers of the form name@domain. Registration costs 10–1000 sats and creates a record in SQLite. The /.well-known/nostr.json endpoint then resolves the name to a pubkey — which is what NIP-05-compatible Nostr clients use to verify identity.

# Resolve alice@bitagent-production.up.railway.app
GET /.well-known/nostr.json?name=alice

{
  "names": { "alice": "npub_hex..." },
  "relays": { "npub_hex...": ["wss://relay.damus.io"] }
}

Trust scoring

Each registered pubkey accumulates a trust score in [0.0, 1.0]. The score is deterministic and factors in payment history, account age, and activity. It's used as a gate for streamfinder.search when IDENTITY_REQUIRE_FOR_PAID_SERVICES=true.

# Get trust signal for a pubkey
POST /a2a
{
  "method": "identity.get_trust_signal",
  "params": { "pubkey": "<npub_hex>" }
}

200
{ "trust_score": 0.72, "warning": null }
