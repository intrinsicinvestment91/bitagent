Payments

Two independent payment paths: Lightning Network (LNbits) and Fedimint ecash.

Two payment paths

Agents accept both Lightning invoices and Fedimint ecash notes. The paths are completely independent — a client can use either without the other being configured.

Lightning

⚡ LNbits

The primary payment path. Every agent endpoint returns a bolt11 invoice on a 402 response. The client pays, then resubmits with the payment_hash.

  • Real payments verified in production
  • Live wallet: ~68k sats
  • Backed by LNbits REST API
  • Requires LNBITS_URL + LNBITS_API_KEY
Ecash

🍃 Fedimint

An additive second path. Agents accept ecash_notes alongside payment_hash. The notes are validated non-destructively first, then reissued into the federation wallet.

  • Live in production alongside Lightning
  • Backed by fedimint-clientd HTTP API
  • Requires FEDIMINT_CLIENTD_URL + password
  • Gracefully disabled if not configured

Lightning payment flow

1

Client sends request without payment

Any POST to a paid endpoint without payment_hash returns HTTP 402.

2

Agent issues a Lightning invoice

402 response includes a bolt11 invoice and the amount_sats. If Fedimint is enabled, also includes ecash_accepted: true and ecash_amount_msats.

3

Client pays the invoice

Any Lightning wallet can pay. The payment_hash is derived from the bolt11 string.

4

Client resubmits with payment_hash

The agent calls AgentWallet.check_invoice(payment_hash). On success it runs the service and returns the result with cost_sats.

# Step 1 — no payment_hash → 402
curl -X POST https://bitagent-production.up.railway.app/search/search \
  -d '{"query": "bitcoin"}'

HTTP 402
{
  "invoice": "lnbc100n1p4q3ala...",
  "amount_sats": 10,
  "ecash_accepted": true,
  "ecash_amount_msats": 10000
}

# Step 2 — pay, then resubmit
curl -X POST https://bitagent-production.up.railway.app/search/search \
  -d '{
    "query": "bitcoin",
    "payment_hash": "1b741d85..."
  }'

HTTP 200
{ "results": [...], "cost_sats": 10 }

Fedimint ecash flow

The ecash path uses a validate → reissue two-step to avoid double-spend without sacrificing speed.

1

Client sends ecash_notes

Include ecash_notes in the request body instead of (or alongside) payment_hash.

2

Agent validates (non-destructive)

FedimintWallet.validate(notes, amount_msats) checks the notes against the federation without redeeming them. Returns True/False.

3

Agent reissues (destructive)

On successful validation, FedimintWallet.reissue(notes) redeems the notes into the federation wallet. This is the point of no return.

4

Service is run

Same result as Lightning path. Response includes cost_sats.

Wallet tiers

WalletLocationRequired configUsed by
_SimWallet src/agents/base_agent.py None BaseAgent subclasses, examples, tests — in-memory, no real payments
AgentWallet agent_wallet.py LNBITS_URL, LNBITS_API_KEY All live agents — lazy loaded, raises if config missing
FedimintWallet src/wallets/fedimint_wallet.py FEDIMINT_CLIENTD_URL, FEDIMINT_CLIENTD_PASSWORD All live agents — eagerly instantiated, disabled gracefully if unconfigured
Important: The @require_payment decorator in src/core/payment.py uses an incompatible escrow/PaymentSecurityManager system and is not wired to the real LNbits wallet. Do not use it. All live agents implement payment checks inline.

Environment variables

# Required for Lightning payments
LNBITS_URL=https://your-lnbits-instance.com
LNBITS_API_KEY=your_lnbits_invoice_key

# Optional — Fedimint ecash (second payment path)
FEDIMINT_CLIENTD_URL=http://localhost:3333
FEDIMINT_CLIENTD_PASSWORD=your_password
